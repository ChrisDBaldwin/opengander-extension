# Contact Protocol Specification

Version 1.0

## Overview

The Contact Protocol is a browser-side handshake that lets a website share a copy of its analytics telemetry with the user. The site proposes contact, the user's agent accepts or declines, and on acceptance, telemetry spans flow from the page to the agent over a BroadcastChannel.

All events use the DOM `CustomEvent` API. All parties run in the same browsing context.

## Roles

| Role | Description |
|------|-------------|
| **SDK** | JavaScript running on the page (site-controlled). Proposes contact, sends spans. |
| **Agent** | JavaScript injected by the user's browser extension. Accepts or declines proposals, receives spans. |

## Window Signal

Before any handshake can occur, the Agent announces its presence:

```js
window.__contact = { version: 1, ready: true }
```

| Field | Type | Description |
|-------|------|-------------|
| `version` | `number` | Protocol version. Currently `1`. |
| `ready` | `boolean` | Always `true`. Indicates the Agent is listening for proposals. |

The SDK polls or checks for `window.__contact` to determine whether an Agent is present. If absent, no handshake is attempted.

This signal is set synchronously when the Agent script loads, before any events are dispatched.

## Proposal

The SDK initiates a handshake by dispatching a `contact:propose` CustomEvent on `document`:

```js
document.dispatchEvent(new CustomEvent('contact:propose', {
  detail: {
    ring: '<ring-name>'
  }
}))
```

### `detail` fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `ring` | `string` | Yes | The data ring being proposed. One of: `traffic`, `interaction`, `identity`. |

A proposal is for a single ring. To propose multiple rings, the SDK dispatches multiple `contact:propose` events, one per ring.

## Response

The Agent responds to a proposal with exactly one of two events, dispatched on `document`:

### Accept

```js
document.dispatchEvent(new CustomEvent('contact:accept', {
  detail: {
    participant_id: '<uuid>'
  }
}))
```

| Field | Type | Description |
|-------|------|-------------|
| `participant_id` | `string` | A UUID (v4) generated by the Agent for this session. Identifies this participant in subsequent span data. |

### Decline

```js
document.dispatchEvent(new CustomEvent('contact:decline'))
```

No `detail` is provided. The SDK must not open a span channel after receiving a decline.

## Decision Logic

The Agent decides based on the user's ring preferences:

| Preference | Behavior |
|------------|----------|
| `always` | Accept immediately. |
| `ask` | Prompt the user (implementation-defined). |
| `never` | Decline immediately. |

The site never learns what preference is set. It only observes `contact:accept` or `contact:decline`.

## Ring Model

Data is categorized into three concentric rings. Each ring is proposed independently.

### Traffic (outermost)

Anonymous navigation data:

- Page views (URL, timestamp)
- Navigation events (route changes)
- Referrer / traffic source
- Web Vitals (LCP, FID, CLS, etc.)

### Interaction (middle)

Behavioral data within a session:

- Click events (target element, coordinates)
- Form submissions (field names, not values)
- JavaScript errors (message, stack)
- Scroll depth, time on page

### Identity (innermost)

Personally-identifying data:

- User IDs
- Custom user attributes (name, email, plan, etc.)
- Cross-session linking

Each ring includes all data from outer rings. Accepting `interaction` implies `traffic` data is also included. Accepting `identity` implies both `traffic` and `interaction`.

The user's preference for each ring is independent. Setting `identity` to `never` and `traffic` to `always` means: accept traffic proposals, decline identity proposals.

## Span Delivery

After the Agent accepts a proposal, the SDK delivers telemetry spans via a `BroadcastChannel`.

### Channel

```js
const channel = new BroadcastChannel('contact:spans')
```

Both the SDK and the Agent open a channel with the name `contact:spans`. The SDK posts messages; the Agent listens.

### Message Format

Each message on the channel is an OTLP-shaped span payload:

```js
channel.postMessage({
  payload: {
    resourceSpans: [{
      scopeSpans: [{
        spans: [{
          name: '<span-name>',
          attributes: [
            { key: 'session.id', value: { stringValue: '<session-id>' } },
            // ... additional attributes
          ],
          startTimeUnixNano: '<nanosecond-timestamp>',
          endTimeUnixNano: '<nanosecond-timestamp>'
        }]
      }]
    }]
  }
})
```

The payload follows the [OTLP/JSON Trace specification](https://opentelemetry.io/docs/specs/otlp/#otlpjson). The Agent must not assume any particular set of attributes beyond `session.id`.

### Required Attributes

| Attribute | Type | Description |
|-----------|------|-------------|
| `session.id` | `stringValue` | Groups spans into sessions. The Agent uses this to synthesize session records. |

All other attributes are optional and depend on the ring and the SDK's instrumentation.

### Lifecycle

1. The SDK opens the BroadcastChannel after receiving `contact:accept`.
2. Spans are posted as they are generated, for the duration of the page session.
3. On page unload (`pagehide`), both sides close the channel.
4. A new page load requires a new proposal/accept cycle.

## Participant ID

The `participant_id` is a UUID v4 generated by the Agent at accept time.

- It is unique per acceptance (per page load, per ring).
- It is returned to the SDK in the `contact:accept` event detail.
- The SDK may include it in span attributes to correlate the user's copy with the site's copy.
- It is not persisted across page loads. Each new acceptance generates a new UUID.
- The Agent may use it internally for session tracking but is not required to.


## Error Handling

- If `window.__contact` is not set, the SDK must not dispatch `contact:propose`.
- If the Agent declines, the SDK must not open a BroadcastChannel.
- If the BroadcastChannel is unavailable (e.g., older browser), the Agent logs a warning and the acceptance still stands, but no spans are delivered.
- Malformed span payloads (missing `resourceSpans` structure) are silently dropped by the Agent.

## Security Considerations

- The Agent runs in the page's MAIN world and is therefore accessible to page scripts. It does not expose any privileged APIs.
- `window.postMessage` messages between inject.js and the content script are filtered by `event.source === window` and by message `type` field.
- Span data is treated as untrusted input. The Agent stores it but does not execute or eval any part of it.
- The `participant_id` is ephemeral and reveals no information about the user.
